# writer-dna — 写作风格基因提取器

> 从你的原始素材中提炼个人写作风格，注入经过验证的 skill 骨架模板，生成专属于你的公众号写作 skill。

**核心理念：框架是 khazix 的，灵魂是你自己的。**

## 项目结构

```
writer-dna/
├── CLAUDE.md              ← 你正在读的这个文件
├── templates/             ← 基础模板（骨架 + 参考示例，不要修改）
│   └── khazix-writer/
│       ├── SKILL.md       ←   通用骨架（风格无关的写作框架）
│       └── references/    ←   参考文件
│           ├── style_examples.md
│           ├── content_methodology.md
│           └── scenes/    ←   场景插件示例（按文章原型组织）
├── raw/                   ← 你的原始素材（放进来，AI 只读不改）
│   ├── profile.md         ←   基础信息（AI 引导生成）
│   ├── chats/             ←   聊天记录、朋友圈截图
│   ├── articles/          ←   你自己写的文章
│   ├── references/        ←   你喜欢的作者的文章
│   └── interviews/        ←   访谈、演讲文字稿
├── wiki/                  ← AI 分析后生成的结构化风格知识
│   ├── persona/           ←   人格轨（我是谁）
│   │   ├── core-values.md ←     核心价值观与定位
│   │   └── tone-voice.md  ←     语气、句式、节奏
│   └── craft/             ←   技法轨（我怎么写）
│       ├── vocabulary.md  ←     词库（高频词、口癖、禁用/推荐词）
│       ├── techniques.md  ←     写作技法模式
│       └── self-check.md  ←     个性化自检体系（含 L5 纠错记录）
└── outputs/               ← 最终生成的 skill 包
    └── <你的账号名>-writer/
        ├── SKILL.md
        └── references/
            └── scenes/    ←   用户专属场景插件（生成时自动创建）
```

## 阶段一：引导收集基础信息

当用户调用 `/writer-dna` 或表达创建写作 skill 的意图时，按以下流程引导。

### Step 1: 收集账号信息

逐个询问（一次一个问题）：

1. "你的公众号叫什么名字？"
2. "用一句话描述你的账号定位。比如'激发大家对AI的好奇'、'让普通人听懂前沿科技'。"
3. "简单介绍一下你自己。你从事什么行业？做了多久？你的读者主要是谁？"
4. "你的 skill 英文代称用什么名字？最终会生成 `<英文名>-writer` 作为 skill 包的目录名和 SKILL.md 的 name 字段。"

### Step 2: 生成 profile.md

将用户的回答整理为 `raw/profile.md`，格式：

```markdown
# 写作风格档案

## 基本信息
- **账号名**：<用户回答>
- **skill英文名**：<用户回答>（用于目录名和 SKILL.md name 字段，如 `entropy-writer`）
- **一句话定位**：<用户回答>
- **自我简介**：<用户回答>

## 补充信息
- **行业背景**：<从简介中提取>
- **目标读者**：<从简介中提取>
- **写作平台**：公众号（当前版本仅支持公众号）
```

### Step 3: 引导放入素材

先判断用户属于哪种情况，再针对性引导：

**情况 A：有自己文章的写作者**
- "你写过公众号文章吗？如果有，把它们保存为 md 或 txt 放到 `raw/articles/`。文章越多，风格分析越精准。"
- "你平时跟朋友聊天有没有特别有意思的记录？放到 `raw/chats/`。聊天记录最能反映你真实的说话方式。"
- "有没有你特别喜欢的作者？觉得'我想写出那种感觉'的？放到 `raw/references/`，我会分析他们的风格供你借鉴。"

**情况 B：纯新手（想学某种写作风格）**
- "你有没有特别喜欢的作者？想写出他们那种感觉的文章？把他们最好的文章放到 `raw/references/`。这是最重要的素材，我会从中学习写作技法。"
- "如果你有朋友圈、微博、知乎回答等能体现你表达方式的素材，可以放到 `raw/chats/`。这不是必需的——对初学者来说，先从模仿开始是完全正常的，你自己的写作声音会在实践中自然形成。"
- "以后你开始写文章了，把它们放到 `raw/articles/`，系统会自动对比你的写作和学习目标的差距，帮你精准调整。"

**通用补充**：
- **raw/interviews/** — "访谈、播客文字稿、演讲记录放到这里。"
- 告诉用户："素材可以分多轮放入，随时补充。放好后告诉我，我就开始分析。"

### 两种用户画像

系统会根据素材情况自动判断用户类型，采用不同的分析策略：

| 维度 | A：有自己文章的写作者 | B：纯新手（学风格） |
|------|---------------------|-------------------|
| 核心输入 | raw/articles/（自己的文章） | raw/references/（想学的作者） |
| 个人声音来源 | articles/（主）+ chats/（辅） | profile.md（定位）+ chats/（可选） |
| 技法来源 | articles/（自己的技法） | references/（学习目标技法） |
| 生成目标 | "让 AI 用我的风格写" | "让我学会 XX 的风格" |
| 技法标注 | 更多 [CONFIRMED] | 技法标 [LEARNED]，待验证 |
| 迭代路径 | 补充文章 → 微调风格 | 开始写文章 → 自己的声音自然浮现 → 过渡到 A |

情况 B 的用户开始写文章后，随着 raw/articles/ 积累，系统会自动从 B 过渡到 A。

## 阶段二：风格分析与 wiki 生成

当用户表示素材已放好，开始分析。

### 双轨分析

分析分两条轨道独立进行，最后合并到 SKILL.md：

- **人格轨（persona/）**：从素材中提炼"你是什么样的人"——核心价值观、语气节奏、情绪表达方式
- **技法轨（craft/）**：从素材中提炼"你怎么写文章"——词库、写作技法、自检规则

几乎所有素材都同时影响两条轨道（聊天记录既反映说话方式也反映叙事习惯）。
分轨的意义在于**分析视角**的分离，不是素材的物理隔离。AI 读取全部素材，
但在生成 wiki 时分两个视角提炼。

### 分析前检查

1. 统计 raw/ 中的素材数量和类型
2. 确定分析深度等级：

| 条件 | 深度 |
|------|------|
| 只有 profile.md | 浅层 |
| profile + 1-5 篇 references（无自己文章） | 中层（技法标 [LEARNED]） |
| profile + 1-5 篇自己的文章 | 中层 |
| profile + 10+ 篇 references 或 >5万字 | 深层（技法标 [LEARNED]） |
| profile + 10+ 篇自己的文章或 >5万字 + 聊天记录 | 深层 |
| 其他混合场景 | 中高层 |

告诉用户当前深度等级和可提升方向（如"更多 references 文章会让技法分析更精准"）。

### 素材优先级

权重根据用户类型动态调整：

**情况 A（有自己文章）**：

| 素材类型 | 权重 | 分析角色 |
|---------|------|---------|
| raw/chats/ | 3x | 真实说话方式（人格轨主输入） |
| raw/articles/ | 2x | 自己的技法偏好（技法轨主输入） |
| raw/references/ | 1x | 审美偏好参考，补充技法灵感 |
| raw/interviews/ | 1x | 场景特殊，有参考价值 |

**情况 B（无自己文章，纯新手）**：

| 素材类型 | 权重 | 分析角色 |
|---------|------|---------|
| raw/references/ | 2x（技法轨） | 写作技法学习源（主要输入） |
| raw/chats/ | 可选 | 朋友圈/微博等可作为补充参考，但非必需 |
| raw/interviews/ | 1x | 场景特殊，有参考价值 |
| raw/articles/ | — | 不存在，提示用户开始写后补充 |

**判断规则**：raw/articles/ 中有文章（≥1 篇）则为情况 A，否则为情况 B。

### 冲突解决规则

当不同素材的风格信号矛盾时：

- **聊天记录 vs 文章**（正式度冲突）：skill 输出以文章风格为基准，vocabulary.md 中记录聊天口语作为"非正式模式"备选
- **情况 A：references vs 自己的风格**：始终优先用户自己的风格，references 仅作灵感补充
- **情况 B：references 内部风格不一致**：如果 references 包含多个作者，标注各作者的技法差异，让用户选择想学谁的。如果只有一个作者，以该作者风格为技法基准
- **早期 vs 近期文章**：默认反映最近 6 个月风格，wiki 中注明演变趋势
- 所有矛盾标记出来让用户决定

### wiki/ 写作规范

每个 wiki 文件必须：
- 以一段摘要开头（3-5 句话概括该文件的核心发现）
- 使用清晰的分节标题，与 khazix-writer SKILL.md 的结构对齐
- 在相关条目之间加入交叉引用，使用带轨道前缀的路径（如"参见 persona/tone-voice.md#情绪表达"、"参见 craft/techniques.md#开头模式"）
- 技法标注使用三档标签：[CONFIRMED]、[SUGGESTED]、[UNIQUE]

### wiki/persona/core-values.md 生成规则

从 raw/profile.md + raw/chats/ + raw/articles/（如有）中提炼：

1. **核心价值观**：
   - 情况 A：从文章中反复出现的观点、判断、好恶中提炼 3-5 个
   - 情况 B：从 profile.md 的定位中提炼 2-3 个；如果 references 中的作者有明确价值观且用户表达了认同，可作为"想践行的价值观"标 [LEARNED]
   - 每个价值观用一句话表述，附带 1-2 个来自素材的证据引用
2. **一句话人设定位**：参考 khazix 的"有见识的普通人在认真聊一件打动他的事"格式，为用户提炼一句。
3. **使命/驱动**：这个账号为什么存在？作者想通过写作实现什么？
4. **反对什么**：明确作者不认同、不做的方向。

### wiki/persona/tone-voice.md 生成规则

从 raw/chats/（如有）+ raw/articles/（如有）+ raw/references/ 中分析：

- 情况 A：从自己的文章和聊天记录中提炼说话方式
- 情况 B（无 chats）：从 references/ 中提炼目标作者的语气和节奏（标 [LEARNED]），从 profile.md 获取用户的基本定位作为人格基础。生成的 tone-voice 会更接近 references 作者的风格，这是正常的——初学者从模仿开始，自己的写作声音会在实践中自然发展

1. **句子节奏**：平均句长、长短交替模式、是否有意制造停顿
2. **段落习惯**：一句话成段 vs 长段落，段落长度分布
3. **人称使用**："我"的使用频率和方式、"你"的称呼方式（"屏幕前的你" / "各位" / "大家"）
4. **情绪表达**：直接 vs 克制、自嘲频率和方式、感叹号/问号/省略号频率
5. **标点偏好**：哪些标点用得多、哪些几乎不用
6. **叙述视角**：私人视角（"我自己也..."）vs 旁观视角、第一人称 vs 第三人称
7. **正式度**：口语化程度、是否使用书面表达

### wiki/craft/vocabulary.md 生成规则

从全部 raw 素材中提取，按用户类型区分分析策略：

1. **高频口语词**：
   - 情况 A：从自己的文章和聊天记录中提取出现 3 次以上的非正式表达
   - 情况 B（无 chats）：从 references/ 中提取目标作者的口语词（标 [LEARNED]，作为学习目标）
   - 分类整理：转场类、判断类、情绪类、拉近距离类
2. **口头禅/口癖**：
   - 情况 A：从自己的素材中提取个人标志性表达
   - 情况 B（无 chats）：从 references/ 中提取作者口癖（标 [LEARNED]）
3. **AI 味禁用词**：
   - 情况 A：用户素材中从未出现的"书面化/AI 常用"表达
   - 情况 B：references 作者也从不使用的表达 + 用户自己不用的表达
4. **推荐替代词**：
   - 情况 A：用户习惯用什么替代正式表达
   - 情况 B：references 作者的替代方式（标 [LEARNED]）+ 用户自己的口语替代
5. **禁用标点**：用户几乎不用的标点（如冒号、破折号）

### wiki/craft/techniques.md 生成规则

从 raw/articles/ + raw/references/ + raw/chats/ 中分析：

1. **文章原型分类**：
   - 情况 A：用户写什么类型的文章？对照 khazix 的 5 种原型，识别用户的类型，补充独有的类型
   - 情况 B：references 中的作者写什么类型？识别用户想学的类型，补充用户感兴趣但 references 中没有的类型（标 [SUGGESTED]）
2. **开头模式**：
   - 情况 A：用户习惯用什么方式开头？每个模式附 1-2 个真实示例
   - 情况 B：从 references 中提取作者的开头模式，附真实示例（标 [LEARNED]）
3. **结尾模式**：引用收尾？哲思留白？行动呼吁？附真实示例。情况 B 标 [LEARNED]。
4. **转场习惯**：用户/目标作者用什么方式衔接段落？
5. **叙事结构**：线性？倒叙？跳跃？英雄之旅？
6. **文化升维参考域**：用户/目标作者习惯连接到什么领域？
7. **特殊技法**：用户独有的写法（khazix 技法列表中没有的）。
8. **每个技法标注**：
   - [CONFIRMED] — 从用户自己的素材中有明确证据
   - [LEARNED] — 从 references 中学习到的技法（情况 B 专属，用户尚未实践验证）
   - [SUGGESTED] — khazix-writer 有但素材中未出现，建议尝试
   - [UNIQUE] — 用户独有，khazix-writer 中没有

### wiki/self-check.md 生成规则

基于其他 wiki 文件生成个性化自检体系。生成原则：**L1/L5 是硬性底线用指令式，L2-L4 是风格偏好用建议式。**

整体通过判定：L1 和 L5 必须全部通过（零命中/零违规），否则阻断输出。L2-L4 不阻断，但在质检报告中标注偏离程度（"达标"/"略偏离"/"严重漂移"）。

**L1 硬性规则**（自动扫描，指令式，零容忍）：
- 从 wiki/craft/vocabulary.md（平铺模式：vocabulary.md）提取禁用词列表和禁用标点
- 规则格式：`指令：全文搜索xxx，出现则必须替换`
- 从 wiki/craft/vocabulary.md 提取空泛表述检查规则
- 定义通过标准：零命中，未通过则必须修复

**L2 风格一致性**（模式匹配，建议式，不阻断）：
- 从 wiki/persona/tone-voice.md（平铺模式：tone-voice.md）提取开头检查规则
- 从 wiki/persona/tone-voice.md 提取节奏和结构检查规则
- 从 wiki/craft/vocabulary.md 提取口语化检查规则（最低词组数量）
- 规则格式：`建议：xxx。参考范围：xxx`
- 定义通过标准：建议达标，未达标标注"风格偏离风险"
- 例外：L2 的标点禁令二次确认保留指令式（本质是 L1 的防回归检查）

**L3 内容质量**（深度审查，建议式，不阻断）：
- 从 wiki/craft/techniques.md（平铺模式：techniques.md）提取观点支撑检查规则
- 从 wiki/craft/techniques.md 提取知识输出方式检查规则
- 从 wiki/craft/techniques.md 提取文化升维检查规则
- 从 wiki/craft/techniques.md 提取文章类型专项检查规则
- 规则格式：`建议：xxx。参考范围：xxx`
- 定义通过标准：建议达标，未达标标注"内容质量建议"

**L4 活人感**（终审，建议式，不阻断）：
- 从 wiki/persona/core-values.md（平铺模式：core-values.md）提取温度感判断标准
- 从 wiki/persona/core-values.md 提取独特性判断标准
- 从 wiki/persona/tone-voice.md 提取姿态判断标准
- 规则格式：`建议：xxx`
- 定义通过标准：整体感知达标，未达标标注"活人感不足"

**L5 运行时纠错**（动态最高优先级）：
- 从对话中收集用户纠错反馈，格式为：`- [日期] 当[场景]时，不应[错误行为]，应[正确行为]`
- 每条 L5 记录必须包含"当[xxx场景]时"的条件，只在匹配的场景下生效
- L5 不能 blanket 否定整条 L1 规则——如果一条 L5 会使某条 L1 在所有场景下失效，必须警告用户
- L5 最多保留 50 条，超出时合并语义相近的记录（保留最新的）
- 相同场景的新纠错覆盖旧纠错（旧记录标记为 `[已替代]`）
- 优先级：L5 > L1 > L2 > L3 > L4
- 定义通过标准：无 L5 记录时自动通过；有 L5 记录时，匹配场景必须遵循纠错指令

## 阶段二（续）：风格确认

wiki 生成后，逐项跟用户确认。一次一个维度，一次一个问题：

**确认示例：**
- "我注意到你写文章时经常用'xxx'，这是你的口癖吗？"
- "你的文章开头好像习惯从xxx切入，我总结了 3 种模式，你看准不准？"
- "这些词我标记为禁用词（你素材中从没用过），要调整吗？"
- "你好像从来没有用过小标题，你的文章都是一口气顺下来的，对吗？"

用户确认后更新 wiki，修正后进入阶段三。

## 阶段三：生成 skill 包

### 生成公式

```
outputs/<账号名>-writer/ = templates/khazix-writer/（骨架）+ wiki/*（风格数据）+ scenes/（场景插件）的融合
```

### SKILL.md 映射规则

> **Important:** 以下表格中的段落名必须与 `templates/khazix-writer/SKILL.md` 的实际 markdown 标题一致。生成前先读取模板文件，校对段落名，如有差异则调整表格。

读取 `templates/khazix-writer/SKILL.md`，按以下规则生成 `outputs/<账号名>-writer/SKILL.md`：

### references/style_examples.md 生成规则

基于 `templates/khazix-writer/references/style_examples.md` 的 19 类分类结构：

1. 从 raw 素材中提取用户自己的真实片段作为示例
2. 素材不够时，AI 根据 wiki 风格规则生成仿写示例，标注"[AI 仿写，请替换为真实素材]"
3. 分类保留规则：
   - 有真实素材 → 保留，填入真实示例，标 [CONFIRMED]
   - 无真实素材但模板有该类别 → 保留，填入仿写示例，标 [SUGGESTED]
   - 用户确认不需要 → 删除
   - wiki/techniques.md 中发现新类别 → 新增，标 [UNIQUE]

### references/content_methodology.md 生成规则

基于 `templates/khazix-writer/references/content_methodology.md` 的结构：

- 选题交集模型保留
- 选题来源根据用户实际情况调整
- HKR 质检法保留
- 过往爆款案例替换为用户自己的文章
- 内容节奏部分用 wiki/tone-voice.md 的发现替换
- 创意案例工作法保留框架，示例替换

### references/scenes/ 生成规则

场景插件是骨架模板与用户风格的桥梁。骨架 SKILL.md 定义通用写作框架，场景插件提供具体文章类型的写作技法。

**生成时机**：在生成 SKILL.md 之后，基于 wiki/craft/techniques.md 的文章原型分类生成。

**生成规则**：

1. 从 wiki/craft/techniques.md 提取文章原型列表
2. 每种文章原型生成一个场景插件文件，文件名用原型名（如 `调查实验型.md`）
3. 每个场景插件包含：
   - 该类型文章的写法重心和注意事项
   - 适用的开头模式（附用户真实示例或仿写示例）
   - 适用的结构模板（附真实示例或仿写示例）
   - 适用的收尾方式
   - 该类型独有的技法（标 [CONFIRMED]/[SUGGESTED]/[UNIQUE]）
4. 通用技法（适用于所有类型的开头、结构、收尾模式）放在一个 `通用技法.md` 文件中
5. 如果用户没有某种技法（如逐一展示法），对应内容标 [SUGGESTED] 或省略

**参考**：`templates/khazix-writer/references/scenes/scene_examples.md` 是 khazix 的场景插件示例，展示了完整的场景插件应该包含什么内容。

### 生成后

告诉用户：
1. skill 包已生成到 `outputs/<账号名>-writer/`
2. 建议先审阅 SKILL.md，重点检查核心价值观、禁用词、风格示例是否准确
3. 建议审阅 `references/scenes/` 下的场景插件，确认文章类型分类和技法是否准确
4. 安装方式：将 outputs 下的整个目录复制到 skills 目录（macOS/Linux: `~/.claude/skills/`，Windows: `%USERPROFILE%\.claude\skills\`）
5. 后续可以随时补充素材到 raw/，重新运行分析来迭代升级

## 迭代升级

当用户补充新素材或要求更新时：

1. 读取新放入 raw/ 的素材
2. 对比现有 wiki，只在有新证据时提出修改
3. 默认反映用户最近的风格（近 6 个月权重更高）
4. 修改以 diff 形式呈现给用户审批
5. 用户确认后更新 wiki，重新生成 outputs

### 对话纠错流程

当用户在写作过程中说"这句不像我"、"这里不对"时，按以下 5 步处理：

1. **理解纠错**：AI 提取 [场景, 错误行为, 正确行为]
2. **追问澄清**：如果提取不明确（用户只说"不对"没说哪不对），追问一次
3. **冲突检查**：检查是否与现有 L5 记录冲突或语义重复，冲突时告知用户
4. **确认写入**：展示将要写入的纠错记录，请用户确认
5. **持久化**：用户确认后追加到 wiki/craft/self-check.md（双轨）或 wiki/self-check.md（平铺）的 L5 纠错记录区，仅更新 outputs 中 SKILL.md 的 L5 部分（不重新生成整个文件）

wiki 单文件超过 500 行时，提示用户是否需要拆分。

## 边界情况

| 情况 | 处理 |
|------|------|
| 只有 profile，无素材 | 降级生成，技法全标 [SUGGESTED]，提示补充素材 |
| 素材是英文/中英混合 | 用素材原始语言分析，skill 保持用户主语言 |
| 用户风格跟 khazix 完全相反 | 不影响，替换范围覆盖整个风格层 |
| 用户想换基础模板 | 修改本文件头部的 base_template 配置，重新生成 outputs（wiki 无需重新分析） |
| 只有 references 无自己的文章 | 按情况 B 处理：references 作为技法学习源（标 [LEARNED]），profile.md 提供基本定位，chats/ 可选。生成的 skill 以学习 references 风格为主，提示用户开始写文章后补充 articles/ 以发展个人风格 |
| wiki/ 既有平铺文件又有 persona/craft/ 子目录 | 子目录优先，平铺文件视为遗留数据，提示用户可删除 |
